# Agent 学习项目 — 工作记忆

## 项目说明
长期学习项目:系统学习 AI Agent 领域。用户是技术背景从业者,关注最新动向与最佳实践。本文件是项目的**持久规则记忆**,每次工作前必须先完整读一遍;**具体的会话进度记录在项目根目录的 `PROGRESS.md`**,开工前一并扫一眼最近几条,确定上次停在哪。

## 产出物
1. notes/ —— 中文。每篇资料一份独立的**全面学习笔记**,首要功能是**帮助理解和复习**——读者通过笔记应能掌握原文所有重要内容,不需要回原文。文件名按 `NN-slug.md`(NN 是两位数专栏序号,slug 与对应 resources/ 文件的核心 slug 一致;例:`notes/01-building-effective-agents.md` 对应 `resources/2024-12-anthropic-building-effective-agents.md`)。**学习笔记是工作主体并持续迭代**;后续可在此基础上派生发表用的专栏文章(专栏名 The Agent Loop),但媒体文章是派生,不是主体。
2. README.md(根目录)—— 英文。这是公开仓库 **agent-atlas** 的对外门面 README。内容包括:项目一句话英文定位、已发期号表(链接到对应 `notes/NN-*.md`)、仓库结构说明、以及学习路线图(随期号积累而增长——精选资料按由浅入深分阶段组织,每篇给出链接、简介、以及为什么在这个阶段读)。第一行用 `# agent-atlas` 作标题。
3. resources/ —— 完整保存的原文,纯素材,辅助学习和写 draft。
4. resources/INDEX.md —— 信息源头地图,记录值得持续追踪的信息源、各源的挖掘状态、以及待探索的线索。

## 每次会话的固定流程
- 开始:先读本文件,再读 `PROGRESS.md` 最近几条,明确上次停在哪、这次用户要什么。
- 结束:在 `PROGRESS.md` **顶部追加新一条**(格式见「### PROGRESS.md 进度区」),然后 **commit + push 到 GitHub**(见「## GitHub 同步」),最后明确告诉用户你停在哪、在等什么决定。
- 任何时候产出草稿或完成一轮修改后都停下等用户确认,不要自动跨越节点。

## 工作流
1. 资料进入:两种来源——用户直接提供链接/文件,或你搜索后给出候选清单由用户筛选。听用户当次指示,不要默认某一种。
2. 搜索候选清单的格式:编号列表,每条含【标题】【来源/作者】【一句话简介】【推荐理由:为什么值得读】,让用户能直接回复编号(如"1、3、5")来选。
3. 保存原文:每确定一篇资料,抓取其完整正文,存为 resources/ 下的 .md 文件(命名与元数据头格式见下)。
4. 更新源头地图:在 resources/INDEX.md 记录或更新该资料所属的信息源及状态,补充由它引出的新线索。
5. 迭代笔记:基于每篇 resources/ 原文产出对应的 notes/NN-slug.md(命名与写作风格见格式约定)。每篇独立,互不串联,不在文件顶部维护跨篇大纲。同一篇笔记随讨论持续迭代;用户会对照原文阅读并提问,你据每次讨论优化对应那一份。
6. 配图:需要图时优先自己用 Mermaid 或 SVG 制作,存入 notes/images/;也可上网搜索或从原文下载合适图表(图注标明来源);确需截图而你无法获取时,请用户提供。
7. 维护路线图:一篇资料被认为有价值时,在根目录 `README.md` 的「Issues」表里加一行(链接对应 notes 文件),并在「Roadmap」段里放进合适阶段,附简介和推荐理由。此文件全程增量维护,始终保持"从零到熟悉"的清晰路线。

## 格式约定(每次必须严格一致,避免跨会话漂移)

### resources/ 文件命名
格式:`发布年月-来源-简短标题.md`,全小写,连字符分隔。
例:`2025-09-anthropic-context-engineering.md`、`2024-12-anthropic-building-effective-agents.md`。
发布年月未知时用 `0000`。

### resources/ 文件元数据头
每个原文文件开头加如下区块,其后接正文:
---
title:
author:
published:
url:
fetched:    (抓取日期)
source:     (所属信息源,与 INDEX.md 对应)
tags:
---

### resources/INDEX.md 结构
固定两节:
「信息源」——表格,列:源名称 | 类型(博客/论文集/播客等) | URL | 状态 | 备注。状态取值:未挖掘 / 挖掘中 / 已挖掘。
「待探索线索」——列表,记录由已读资料引出、值得追的新方向。

### PROGRESS.md 进度区
项目根目录的独立日志文件 `PROGRESS.md`,追加式,最新一条在最上。每条格式:
[日期] 做了什么 | 产出或改动了哪些文件 | 当前停在哪、在等用户什么决定。

### notes/ 文件命名与写作风格

**命名**
- `NN-slug.md`,NN 是两位数专栏序号(01、02、…),slug 与对应 resources/ 文件的核心 slug 一致。例:`notes/01-building-effective-agents.md` ↔ `resources/2024-12-anthropic-building-effective-agents.md`。

**核心定位 —— 学习笔记,服务于「理解 + 复习」**
- 笔记必须**全面覆盖**原文重要内容:概念、定义、判断标准、例子、附录细节,**一个不漏**;读者只看笔记应能拿到原文所有重要内容,不必回原文。
- 但**笔记不是仿照原文做翻译**——要按**学习与复习的逻辑重新组织**:同类内容聚合(如 Appendix 2 的工具相关内容并入 ACI 一节)、概览/速查表前置、并行概念用**表格压成对比**而不是分散成多段叙述、把"决策/选型"这类隐含逻辑显式化。
- **覆盖度对原文负责,组织顺序对读者负责**。
- 笔记不是个人评论/导读散文;**作者立场和延伸思考默认不进笔记**,需要时以"备注"短句嵌入相关段落,不另起总结段。

**篇内结构(典型骨架,按内容酌情调整)**
- 每篇独立成文,不与其他篇横向拼接;**不在文件顶部维护跨篇大纲或专栏自我介绍**。
- **开头**:第一行是封面图(`![...](covers/NN-cover.png)`,生成与风格见下「**封面**」),然后一行原文出处(作者/年月/备份路径)+ 必要的版本/时效备注(独立 italic 行)。封面本身已含标题,所以 markdown 里**不再写 H1 标题**。
- **概览(强烈推荐)**:几条 bullet 把核心立场、关键区分、技术栈/概念骨架、核心原则、应用领域压成 TL;DR,作为读者的入口。
- **正文**:按学习逻辑组织(常见排序:关键概念 → 决策与取舍 → 各类模式速查 → 主要对象细节 → 工程原则 → 工具/接口 → 落地领域 → 组合扩展);章节用中文描述性命名(如「一、关键概念」「二、决策与取舍」),不照搬原文英文章节名当伪标题。
- 每节内**优先用表格、列表、对比代替散文**;短句、术语粗体、英文术语保留原词。
- **结尾不堆"批注与提问点 / 讨论种子 / 写在最后"这类元结构段落**;讨论留给会话本身。

**封面**
- 每篇笔记顶部一张封面,**统一风格**(简约杂志风 + Anthropic 暖色调:cream `#F0EEE6` / terracotta `#CC785C` / charcoal `#1F1F1F`;humanist 排版;大量留白;2 色几何抽象图示)。
- **平台与比例**:主战场是**小红书**,默认 **3:4 竖构图(1080×1440)**;脚本可选 `3:4 / 9:16 / 16:9 / 1:1 / 2.35:1`,要发别的平台再切。
- **文字克制**:封面只放两段文字——顶部 `THE AGENT LOOP · ISSUE NN` 小标签 + 一句中文 hook;**没有副标题、没有英文长标题、没有正文**。
- 文件位置:`notes/covers/NN-cover.png`(NN 同笔记序号)。
- 生成方式:`scripts/gen-cover.sh <issue> "<hook-zh>" "<visual_hint>" [ratio]`——脚本里**写死了统一风格 prompt**(含「禁动其他文件」的硬约束),只让 hook、视觉隐喻、可选比例三个变量变化。底层走 Gemini CLI(`gemini -y --skip-trust -p`)。
- 在 markdown 顶部用 `![...](covers/NN-cover.png)` 引用,封面即一期的"标题图",因此 markdown 里**不再写 H1 标题**。

**配图**
- 若原文有图示且能直观体现「分类 / 流程 / 结构」,**优先复用原文图**——下载存到 `notes/images/`,文件名 `NN-slug.png`(NN 同笔记序号,例:`notes/images/01-prompt-chaining.png`)。
- 在 markdown 用 `![alt](images/NN-slug.png)` 引用,**下面紧跟一行小字图注**:`*图:原作者 — 简短说明*`,注明来源 + 一句话点出图里在演示什么。
- 也可以自己用 Mermaid / SVG 制作,放进同目录;原文确无现成图时再制作。

**写作风格 —— 避免「AI 味」**
- 不堆"最有价值的一刀"、"那个被低估的词"、"祛魅"、"翻译成 X 语境"等标题党短语和套路修辞;小标题以**反映内容**为准。
- 加粗用于**术语和关键定义**(服务复习扫读),而不是"每段一个金句"的装饰。
- 标题层级以利于扫读为准;**最深不超过 H3**(必要时 H4 用粗体内联标签代替)。
- 引用原文以**短句和概括**为主,绝不大段复述;翻译关键英文短语时**保留原英文术语**(workflow / agent / ACI / ground truth / sectioning / voting 等),便于读者回原文索引。
- 必要时可加一两处"备注"补充背景或时效信息(如线上版本已更新、与某后续文献的关联),但不要每段都加注脚,也不在篇末做集中总结/反思。

## 内容结构(刻意不提前钉死)
- 单篇笔记按**学习与复习的逻辑组织**(同类聚合、概览前置、并行概念表格化),**不仿照原文章节做翻译**。覆盖度对原文负责,组织顺序对读者负责。
- 根目录 `README.md` 的路线图阶段划分,先用临时骨架起步,等积累了第一批真实资料后,再和用户一起设计正式结构。不要凭空预设一套复杂结构。
- **不维护跨篇全局大纲**;notes/ 目录的文件清单(按 NN 排序)本身就是序列。

## 关键规则
- notes/ 下每篇笔记用中文;根目录 `README.md` 用英文(它是公开 agent-atlas 仓库的对外门面)。
- 写作面向技术从业者:可使用专业术语,聚焦动向与最佳实践,不做入门科普式铺垫。README 是例外——它面向新手,组织上要循序渐进,但单篇资料本身的难度不必回避。
- 尊重版权:笔记和 README 都不大段复制原文,引用简短并注明来源,用自己的话重新组织。resources/ 的全文仅供个人对照学习。
- resources/ 的原文文件不改动,它是原始素材;所有加工产出放在 notes/。
- tmp/ 只放临时文件,不放任何产出物。

## GitHub 同步
- 公开远端仓库:**agent-atlas**(`https://github.com/haosenssss/agent-atlas`),默认分支 `main`。
- **每次会话结束前必须 commit + push**——常规一次会话一个 commit;commit message 概括核心改动(中英文皆可),例:`feat(01): add 8 inline diagrams + xiaohongshu cover` / `chore: move progress log to PROGRESS.md`。
- 命令模板:`git add -A && git commit -m "<msg>" && git push`。
- `.gitignore` 排除项:
  - `resources/` —— 原文备份(版权,仅本地)
  - `tmp/` —— 临时文件 / 抓取中间产物
  - 系统垃圾(`.DS_Store` 等)、编辑器残留(`.vscode/` `.idea/` 等)、Python 虚拟环境、`.env`
- `notes/images/` 里的原文配图(带图注注明来源)正常同步,属学习笔记的引用配图,教育用途 + 出处明示。
- `notes/covers/` 是自己生成的封面,正常同步。
